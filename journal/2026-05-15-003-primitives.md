**Date:** 2026-05-15

**Upstream source:** `.rust-by-example/src/primitives.md` (snapshot 898f0ac)

## What I tried

Port the first chapter outside the "Hello World" section: `primitives.md`, the parent of the primitives subtree (literals, tuples, arrays). New top-level heading `= Primitives` at the same depth as `= Hello World`.

## What happened

The only code block in the chapter is tagged `rust,editable,ignore,mdbook-runnable`. It contains a deliberate type error:

```rust
let mut mutable = 12; // Mutable `i32`
mutable = 21;

// Error! The type of a variable can't be changed.
mutable = true;
```

The chapter teaches that "the type of a variable can't be changed". Sending the snippet to evcxr would surface a compile-error box, which is *not* the pedagogical demonstration upstream is making — the reader is supposed to see the source and the inline comment, not an error sidecar in the rendered output.

Rendered as plain `#raw(..., lang: "rust", block: true)` only, matching the source-only convention previously used in `print_debug.typ` / `print_display.typ` for definitions-only blocks (commit `1ba0906`). A short paragraph above the snippet explains the choice so readers don't wonder why this block isn't evaluated.

Bare `typst compile` succeeds; PDF grows 286 → 317 KB matching the new chapter.

## What I learned

`ignore,mdbook-runnable` upstream snippets are a third reason to render source-only, beyond the two already pinned in `examples/rust-by-example/CLAUDE.md`:

1. Explanatory top-level definitions that would leak across chapters in the shared evcxr context.
2. Activities / "try this yourself" stubs.
3. **(new)** Snippets that deliberately fail to compile because the chapter is *teaching the error*.

Worth folding into the chapter CLAUDE.md the next time the rbe conventions get a touch-up — not done in this PR to keep diff narrow.
