## What I tried

Port upstream `hello.md` by hand to `examples/rust-by-example/hello.typ`, included from a top-level `main.typ` that uses `#evcxr.setup()` and `#outline()`. Goals: (1) bare `typst compile main.typ` succeeds with placeholders (D-004), (2) `evcxr-typst run --allow-eval` evaluates the snippet and the captured stdout `Hello World!` lands in the sidecar, (3) the experience surfaces something worth journaling.

## What happened

Three findings, in the order they hit.

### 1. `#include`'d files don't share the parent's imports

First bare-compile attempt failed:

```
error: unknown variable: evcxr
  ┌─ examples/rust-by-example/hello.typ:9:1
  │
9 │ #evcxr.rust(id: "rbe-hello", ```rust
  │  ^^^^^

help: error occurred while importing this module
   ┌─ examples/rust-by-example/main.typ:23:9
   │
23 │ #include "hello.typ"
```

I had written `#import "../../packages/evcxr/lib.typ" as evcxr` only in `main.typ`, assuming the `#include`'d file would inherit. It does not — Typst's `#include` evaluates the included file with its own scope. Each chapter file needs its own `#import`. After adding the import to `hello.typ`, bare compile succeeded (51.7K PDF; the only output was a benign warning about a missing `DejaVu Sans Mono` font in `fallback.typ`).

This is non-obvious on a first read of the Typst docs because most templating examples use `#import` everywhere, but the `examples/rust-by-example/` setup is inherently `#include`-heavy (it's a book with many chapters, each rendered in turn into the parent document body). Implication for the chapter style: every chapter file is a small island, importing the package itself.

### 2. The fn main wrapper — fidelity vs. evcxr's eval model

Upstream `hello.md` snippet:

```rust
fn main() {
    println!("Hello World!");
}
```

evcxr would *define* `main` and never call it; `Hello World!` would never print. Three possible resolutions, considered:

1. **Drop the wrapper, inline the body.** Most natural for evcxr; loses the upstream "this is `main`" pedagogical framing.
2. **Show the wrapper, append `main();` to the visible source.** Fidelity to upstream; the trailing call is honest about the eval contract.
3. **Wait for `rust-main(...)` (T-B01 / D-022).** Cleanest long-term but blocks any chapter today.

Picked **(1)** for now. Trade-off documented in `examples/rust-by-example/CLAUDE.md` § "Fidelity vs. evcxr's evaluation model". The first chapter that genuinely *needs* `fn main()` semantics (e.g. one demonstrating `argv` parsing) will force option (2) or push T-B01 to the front of the queue.

### 3. SVG export bails on multi-page documents

`evcxr-typst run --allow-eval` evaluated the snippet successfully (1 snippet in 32.3 s cold, 2.0 s warm — `:cache 500` doing its job) and rendered the PDF. Then SVG export failed with:

```
error: cannot export multiple images without a page number template ({p}, {0p}) in the output path

Error: `typst compile` (svg) failed (status exit status: 1)
```

The CLI was set up to render SVG to a single file path next to the entry; multi-page documents (the rbe book is multi-page from page 1 thanks to the outline) reject that. The CLI source already flagged the limitation in a comment at `cli.rs:218` but treated SVG failure as a hard `bail!`, which made the whole `evcxr-typst run` fail even though the actual evaluation succeeded.

**Fix landed in this branch:** `cli.rs` now downgrades SVG render failure to a warning. The PDF is the user-facing artifact; SVG was always documented as the visual quick-look fallback. Multi-page books that want per-page SVGs can invoke `typst compile` directly with a `{p}` template (already documented in the root `CLAUDE.md`).

## What I learned

- **Each chapter file imports the package independently.** Making this implicit in the `examples/rust-by-example/CLAUDE.md` invariants list. (Done.)
- **`fn main()` is a fidelity decision per chapter.** Option (1) is the default until `rust-main` ships; track per-chapter exceptions.
- **The CLI's "render PDF + SVG" assumption was single-page-only.** Rust-by-example breaks that assumption immediately. Downgrade to a warning is the smallest fix that keeps PDF behaviour identical.
- **`:cache 500` cold→warm is real.** 32.3 s → 2.0 s for the same hello snippet. That's the incremental compilation working.
- **rust-analyzer leaks `cargo metadata` warnings to stderr.** Noise during eval; doesn't affect outcome. Worth tracking if it gets louder.
