## What I tried

## What happened

Two real bugs surfaced before the chapter got past scaffolding.

### Bug 1: watch ignored subdirectory edits (#27)

`evcxr-typst watch examples/rust-by-example/main.typ` did not re-evaluate when `hello/comment.typ` was edited. The typst-watch child reported `compiled successfully`, but evcxr never ran a cycle — `_index.json` stayed at the previous chapter's snippet ID list.

Root cause in `crates/evcxr-typst/src/watch.rs` (pre-fix):
- the notify watcher registered `entry.parent()` with `RecursiveMode::NonRecursive`, so events for files under `hello/` never reached us;
- `is_relevant()` accepted only paths whose `.parent()` equalled `entry.parent()`, rejecting any descendant deeper than one level.

Routed through the subagent PR loop (per `feedback_real_bug_subagent_loop.md`): opus planner → sonnet implementer → opus reviewer → opus closer. PR #27 changed the parent watch to `Recursive` and broadened `is_relevant` to `path.starts_with(entry_parent)` (preserving the cache-dir exclusion). New integration test `tests/watch_subdir.rs` pinned the regression. Squash-merged at `23d0a95`.

### Bug 2: `is_relevant` never matched relative entry paths (#28)

After #27 merged and was rebased into the track branch, the authoring loop *still* did not eval on subdir edits. Repro: typst-watch saw the change and recompiled, but no `evcxr_typst::watch: watch cycle plan` debug line ever appeared. Toggling the entry to an absolute path (`$(pwd)/examples/rust-by-example/main.typ`) made cycles fire immediately.

Root cause: notify delivers absolute event paths; `entry_parent` was computed from whatever the user passed on the CLI. `Path::starts_with` is a literal component-prefix test, not a path-resolution match. A relative `entry_parent` never matches an absolute event path.

Why #27's test missed it: `tests/watch_subdir.rs` builds `entry` from `CARGO_MANIFEST_DIR.join(...)`, which is always absolute. The relative-path code path was not exercised.

Routed through the same subagent loop. PR #28 added one line at the top of `watch_thread`: `let entry = std::fs::canonicalize(&entry).map_err(Error::Io)?;`. New `tests/watch_subdir_relative.rs` exercises a relative entry via `set_current_dir` and asserts `_index.json` updates after a *subdir* mutation (not just an entry-dir mutation, per reviewer feedback in PR #28 which prompted commit `93e9f77` from the fixer). Squash-merged at `948d78f`.

End-to-end validation after rebase: edited `hello/comment.typ` while `evcxr-typst watch examples/rust-by-example/main.typ` was running with the relative path. Saw `watch cycle plan plan="reset_and_replay"`, sidecar `rbe-hello-comment.txt` updated to include the live-edit `println!`. The full authoring loop now works for sub-directory chapters with relative CLI paths.

### Snippet content correctness

`evcxr-typst run --allow-eval` produces `Is x 10 or 100? x = 10` for `rbe-hello-comment`, matching upstream's expected output exactly. The nested-block-comment construct (`/* … /* nested */ */`) and the `/* … // */` close both compile and behave as Rust intends. evcxr's eval-context model does not change comment semantics.

## What I learned

- The watch authoring loop is more fragile than the existing tests suggest. The two bugs I hit were each one-line fixes but neither was caught by the merged-as-of-this-morning test suite; both required real CLI use to surface. Future watch tests should cover the CLI invocation shape (relative path from cwd), not just the library-API shape.
- The reviewer subagent correctly distinguished a *vacuous* test from a *narrow* test in PR #28: the as-written test pinned the relative-path bug but did not pin the subdir code path that #27 had introduced. Adding the second mutation step before merge is the kind of catch the loop is supposed to produce.
- `cargo build` did not rebuild after the prior session's `b354b50` watch-fix landed in cli.rs — binary mtime had advanced (something else touched it) but the source hash hadn't. `touch crates/evcxr-typst/src/cli.rs && cargo build` was needed. Pre-existing edge case; not investigating further this session.

## Follow-ups

- [ ] **Watch noop runaway after typst-watch rewrites** — filed as #29. After a real edit-and-cycle finishes, `watch cycle plan plan="noop"` fires every ~660 ms indefinitely. Some typst-watch artifact in the entry's parent directory is tripping `is_relevant` on each iteration. Plausible candidates: `main.pdf` rewrite, or a typst-watch internal poll.
- [ ] **Watch does not backfill missing sidecars on startup** — filed as #30. If `_index.json` is missing entries that `discovery::discover` returns, `classify(prev, curr, _)` still returns `Plan::Noop` (because `prev == curr` after the first cycle copies discovery into `prev_snippets`). The user has to run `evcxr-typst run` once to seed sidecars before `evcxr-typst watch` is useful.
- [ ] **`tests/watch_subdir_relative.rs` red message granularity.** On unpatched code the test fails at step 1 (entry-dir bug also broken) rather than step 2 (subdir-specific). A future regression that breaks *only* subdir delivery would be caught at step 2 with the right diagnostic; this is fine — flagged in the fixer's report. No action needed.
