## What I tried

Port the if-let chapter. Four snippets upstream:

- Awkward-match comparison (bare top-level statements, no `fn main`).
- `if let` with `Option<i32>` and various else branches — has `fn main`.
- `if let` matching `enum Foo { Bar, Baz, Qux(u32) }` — has `fn main`.
- Compile-error challenge (`enum Foo {Bar}` with `if Foo::Bar == a` — tagged `ignore,mdbook-runnable` upstream).

Initial plan: `evcxr.rust(...)` for the bare-statement awkward block, `evcxr.rust-main(...)` for the two `fn main` blocks, and `#raw(..., lang: "rust", block: true)` for the deliberate-compile-error challenge.

## What happened

The two `rust-main` blocks evaluated and matched upstream verbatim:

- option: `Matched 7!` / `Didn't match a number. Let's go with a letter!` / `I don't like letters. Let's go with an emoticon :)!`
- enum: `a is foobar` / `c is 100` / `c is one hundred`

The awkward-match block failed in evcxr with:

> Couldn't automatically determine type of variable `optional`. Please give it an explicit type.

evcxr's persistent top-level scope can't synthesize a concrete type for `let optional = Some(7);` without an explicit annotation — `Option<_>` is unconstrained until the `match` arms type-check, but the binding has to be slot-declared *before* the match. Upstream gets away with it because the snippet lives inside a hypothetical `fn main()` where local inference is fine.

Three options:

1. Wrap the bare statements in a synthetic `fn main()` and use `rust-main`. Deviates from upstream's visible source (upstream specifically shows bare statements) but evaluates cleanly.
2. Add `: Option<i32>` to the `let` (upstream's adjacent comment literally says "Make `optional` of type `Option<i32>`"). Smallest deviation from upstream textually, but still a deviation.
3. Render source-only. Loses the "see, match works here too" evidence but preserves upstream source verbatim and matches existing convention for blocks that would pollute the persistent context (`let optional` would persist forever otherwise).

Picked (3). The block's pedagogical role is comparative ("this is awkward") rather than demonstrative; the reader doesn't need to see output. Added an inline note next to the rendered fence explaining the evcxr top-level inference quirk.

## What I learned

- evcxr's top-level inference needs explicit types for unconstrained generics (`Option`, `Result`, etc.) even when a downstream `match` would fully constrain them — because the binding is allocated before subsequent statements run.
- This is the second class of "snippet that compiles upstream but won't eval in evcxr-typst" we've hit (alongside redefinition collisions). Worth eventually surfacing in the rbe CLAUDE.md as a third explicit "render source-only when…" bullet.
