## What happened

The watch loop started with one transient Typst error because `_index.json` did not exist at the exact moment Typst first compiled. The evaluator produced the index immediately afterward and the next compile succeeded. This is the same cold-cache startup edge noted in the watch-sidecar work: noisy, but not fatal.

The first upstream code block defines `UnPrintable` and `DebugPrintable` only. It evaluates cleanly, but intentionally has no stdout; the useful artifact is the manifest sidecar that proves it ran. Rendering it with `evcxr.rust(...)` exposed a package assumption: successful `rust(...)` snippets were expected to have a `.txt` sidecar. The chapter now renders that block as raw Rust source plus `evcxr.rust-hidden(...)` for evaluation.

The two later blocks keep upstream's `fn main() { ... }` wrapper and now use `evcxr.rust-main(...)`. That keeps `name`, `age`, and `peter` local to the function body while still making evcxr execute the example; the synthetic `main();` call is hidden inside the CLI's execution source, not rendered in the chapter.

After seeing the cold-cache `_index.json` error in watch mode, I fixed the watcher to write an empty available-index before spawning `typst watch`. The first Typst compile now has a valid index and renders placeholders until evaluation materializes real sidecars.

## What I learned

`fmt::Debug` examples are a good stress test for state persistence. Top-level structs such as `Structure` and `Person` persist in evcxr's context and can be shadowed later by name; local values inside `main` do not leak. That means `rust-main(...)` is the right default for examples whose pedagogical source includes `fn main()` plus local bindings.

The first no-stdout snippet also shows a small testing distinction: a successful snippet can have no `.txt` file when it emitted no plain stdout. The manifest sidecar is the stable proof of evaluation for that case.
