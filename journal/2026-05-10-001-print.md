## What happened

Three things worth noting; nothing fundamentally broken.

### 1. Per-chapter fidelity decision: option 2 over option 1

The hello and comment chapters used option 1 (drop the `fn main()` wrapper, inline the body) per `examples/rust-by-example/CLAUDE.md`. This chapter is different in two ways:

- The body declares `let number: f64 = 1.0;` and `let width: usize = 5;` inside `fn main`. Inlining them as top-level evcxr statements would commit `number` and `width` as bindings visible to every subsequent chapter ‚Äî leaking common variable names. evcxr only tracks top-level items; bindings inside a function body don't escape.
- The body declares `struct Structure(i32);` (preserved verbatim because upstream uses it pedagogically with a deliberately-commented `Display`-failure line). Same scoping concern ‚Äî wrapped in `fn main`, the struct stays local to that function and doesn't leak.

So this chapter uses **option 2**: keep `fn main() { ... }` as the visible source, append a synthetic `main();` call at the end of the snippet so evcxr actually invokes it. Both lines are in the rendered source ‚Äî readers see exactly what is being asked of evcxr.

This is the first chapter to deviate from option 1. Updated `examples/rust-by-example/CLAUDE.md` so future chapters know to pick option 2 when there are bindings or items inside the wrapper that shouldn't leak.

### 2. The deliberate FIXME

Upstream marks the snippet `rust,editable,ignore,mdbook-runnable` because of a deliberate FIXME on the line:

```rust
println!("My name is {0}, {1} {0}", "Bond");
// FIXME ^ Add the missing argument: "James"
```

The format string requires two args (`{0}` and `{1}`) but only one is supplied ‚Äî `rustc` rejects it with E0061. Upstream's `mdbook-runnable` plugin shows a Run button that fails on click, inviting the reader to fix it.

We can't show a "fix and re-run" interactive surface, and we want the snippet to actually evaluate so readers see the rich output (binary, octal, hex, padding). Two options considered:

1. Render the snippet as a plain Typst raw block (no `evcxr.rust(...)`) so it's source-only ‚Äî faithful to upstream's `ignore` directive but readers lose the educational output.
2. Fix the FIXME by adding `"James"` so the snippet compiles, document the deviation in this entry and in a comment in the chapter source.

Picked **(2)**. The chapter header and the line-comment in the snippet flag the change explicitly; readers who want to experience the FIXME can revert the line locally. Adding a per-chapter "compile-fail demonstration" in a future port phase would need a different mechanism (T-I07 pretty-error rendering on a separately-id'd snippet); not in scope here.

## What I learned

- The per-chapter fidelity choice is a real design lever, not a footgun. Some chapters genuinely need the `fn main()` wrapper for scope hygiene; others are cleaner without. Writing the chapters by hand surfaces this ‚Äî a deterministic porter would have to know about evcxr's scoping rules to make the same call.
- "Fix the FIXME" is the right move for chapters where the upstream FIXME is incidental to what the chapter teaches. For chapters where the FIXME *is* the lesson (the `error/*` chapters), we'll need a separate evcxr.rust mode that captures the compile error via T-I07's pretty-error rendering. Defer until B5.
- All four bug fixes from journal 002 / 003 (#27, #28, #32, #33) compose correctly ‚Äî the watch loop is now usable for incremental authoring; this chapter was ported `run`-only to keep the entry focused, but the watch path is healthier than it was at the start of the rbe track.
